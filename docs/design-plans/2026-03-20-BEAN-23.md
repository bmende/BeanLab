# LAN DNS Server Design

## Summary

This design adds a LAN-facing DNS server to the BeanLab cluster so that local hostnames like `jellyfin.beanlab` resolve on any device connected to the home network. A standalone CoreDNS instance is deployed as a Kubernetes Deployment in a dedicated `dns` namespace, pinned to the wasabi node and binding `hostPort: 53` so LAN clients can reach it at wasabi's static IP (`192.168.50.101`). The router's DHCP server is reconfigured to hand out wasabi as the primary DNS server and itself as fallback, meaning internet resolution continues working when the cluster is offline — only local names become unavailable.

The design draws a deliberate split between what is managed by Flux and what is managed locally. The Corefile (CoreDNS behavior and plugin configuration) lives in a Kubernetes ConfigMap committed to git and reconciled by Flux like any other infrastructure component. Zone files (the actual DNS records) live on wasabi's filesystem at `/etc/coredns-lan/zones/` and are mounted into the pod via hostPath — they are never committed to git. This means operators edit zone files directly over SSH without touching the cluster's GitOps workflow. CoreDNS's `auto` plugin watches the zone directory and reloads any file named `db.<zone>` automatically when its SOA serial increments, so adding or updating a zone requires no Kubernetes changes at all.

## Definition of Done

1. A LAN-facing DNS server runs in k3s, capable of serving arbitrary local zones (`.beanlab` initially, but not limited to it)
2. DNS records are stored locally on the node (not in git) and managed by editing a file
3. The DNS server deployment itself is Flux-managed (in git) like other infrastructure
4. Router DHCP hands out the DNS server as primary with the router as fallback, so LAN DNS works when the cluster is up and degrades gracefully when it's down
5. The record format supports arbitrary A records (any hostname/zone to any IP), not just k3s services

## Glossary

- **CoreDNS**: A Go-based DNS server used natively inside Kubernetes as the cluster DNS provider. This design deploys a second, independent CoreDNS instance outside of `kube-system` to serve LAN clients rather than pod-internal DNS.
- **Corefile**: CoreDNS's primary configuration file. It declares which plugins are active and how they are ordered for each server block. Stored in a ConfigMap in this design.
- **Zone file**: A plain-text file in RFC 1035 format listing DNS records for a domain (zone). Each record maps a hostname to an IP or other resource. Files in this design follow the naming convention `db.<zone>` (e.g., `db.beanlab`).
- **SOA record (Start of Authority)**: The first record in a zone file. Its serial number field is used by DNS servers to detect whether the zone has changed. CoreDNS's `auto` plugin only reloads a zone when the serial increments.
- **`auto` plugin**: A CoreDNS plugin that watches a directory for zone files and serves any it finds as DNS zones, picking up additions and changes without requiring a server restart or Corefile edit.
- **`forward` plugin**: A CoreDNS plugin that proxies queries that do not match any local zone to upstream resolvers (here, Google's `8.8.8.8` and `8.8.4.4`).
- **hostPort**: A Kubernetes pod field that binds a container port directly to the same port on the node's network interface, making it reachable at the node's IP. Distinct from a NodePort Service, which uses a randomly assigned high port.
- **hostPath**: A Kubernetes volume type that mounts a directory from the node's filesystem into a pod. Used here so zone files at `/etc/coredns-lan/zones/` on wasabi survive pod restarts and are editable via SSH without knowledge of Kubernetes internals.
- **DHCP (Dynamic Host Configuration Protocol)**: The protocol by which a router assigns IP addresses and network settings — including DNS server addresses — to devices when they join the network. Reconfiguring DHCP DNS entries is Phase 3 of this design.
- **DNS failover**: The behavior where a device tries its secondary DNS server when the primary is unreachable. In this design the router acts as the fallback, preserving internet access when the cluster is down.
- **Flux Kustomization**: A Flux CD custom resource that points to a directory of Kubernetes manifests in git and reconciles them to the cluster on a schedule. This design adds `coredns-lan/` as a resource under the existing `infrastructure` Kustomization rather than introducing a new one.
- **`prune: true`**: A Flux Kustomization setting that deletes Kubernetes resources from the cluster when their corresponding manifest is removed from git.
- **kube-dns**: The CoreDNS instance that k3s deploys automatically in `kube-system` for internal cluster DNS (pod-to-service resolution). Entirely separate from the instance this design creates.

## Architecture

A standalone CoreDNS instance runs as a Deployment in the `dns` namespace, pinned to wasabi via `nodeSelector: node-role.beanlab/streaming: "true"`. The pod binds `hostPort: 53` (UDP and TCP) so LAN devices query wasabi's IP (`192.168.50.101`) on the standard DNS port.

**Configuration split:**
- **Corefile** (how CoreDNS behaves) — stored in a ConfigMap, committed to git, Flux-managed
- **Zone files** (DNS records) — stored on wasabi's filesystem at `/etc/coredns-lan/zones/`, mounted into the pod via hostPath, NOT in git

CoreDNS uses the `auto` plugin to scan the zone directory. Each file named `db.<zone>` is automatically served as a zone (e.g., `db.beanlab` serves the `beanlab` zone). Adding a new zone is as simple as dropping a new file — no Corefile changes needed.

Non-local queries are forwarded upstream to `8.8.8.8` and `8.8.4.4` via the `forward` plugin. Responses are cached (30s default for upstream, zone TTL for local records).

**DNS failover:** Router DHCP is configured with DNS Server 1 = `192.168.50.101` (wasabi/CoreDNS) and DNS Server 2 = `192.168.50.1` (router). When the cluster is down, devices fall back to the router for internet DNS resolution. Local `.beanlab` names won't resolve during downtime, but internet access is unaffected.

**Data flow:**
1. LAN device queries `192.168.50.101:53` for `jellyfin.beanlab`
2. CoreDNS `auto` plugin matches the `beanlab` zone, returns the A record from `db.beanlab`
3. LAN device queries for `google.com` — no local zone matches, `forward` plugin sends to upstream, caches the response
4. Cluster is down — device retries with DNS Server 2 (router), internet DNS still works

## Existing Patterns

Investigation found no existing DNS customization in the codebase. The built-in k3s CoreDNS runs in `kube-system` as `kube-dns` (ClusterIP only) — this design is a completely separate instance in a dedicated namespace with no interaction.

**Patterns followed from existing infrastructure:**
- New component in `infrastructure/coredns-lan/` with its own `kustomization.yaml`, added to `infrastructure/kustomization.yaml` (same as `storage/`, `cert-manager/`, `traefik/`, `headlamp/`)
- Flux Kustomization in `clusters/beanlab/infrastructure.yaml` already covers `./infrastructure` path — no new Flux Kustomization needed
- `prune: true` works cleanly because zone files are hostPath content, not Kubernetes resources — Flux only manages the Deployment, ConfigMap, and Namespace

**New pattern introduced:**
- hostPath mount for user-editable data (`/etc/coredns-lan/zones/`). This differs from the existing local-path PVC pattern used by app config. Justification: zone files must exist at a known, stable path on the host for manual editing via SSH. A PVC's actual path on disk is opaque and harder to edit directly.

## Additional Considerations

**SOA serial convention:** Zone files use `YYYYMMDDNN` serial format (e.g., `2026032001`). The `auto` plugin only reloads when the serial increments. Forgetting to bump the serial is the most common operational mistake — the seed zone file will include a comment documenting this convention.

**Security context:** CoreDNS requires only `NET_BIND_SERVICE` capability for port 53. No privileged mode, no other elevated permissions.

**Zone file syntax errors:** CoreDNS logs the error and skips the malformed zone. Other zones and upstream forwarding continue working. No cascading failure.

**Adding new zones:** Drop a file named `db.<zone>` in `/etc/coredns-lan/zones/` on wasabi. The `auto` plugin discovers it automatically within 10 seconds. No Corefile or Kubernetes changes needed.
